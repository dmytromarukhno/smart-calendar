# SPEC.md — Функціональна специфікація

## 1. Опис продукту

Програмний продукт **«Календар для розумного будинку»** — інформаційна система, що поєднує функції планувальника подій з автоматизованим керуванням побутовими IoT-пристроями (освітленням, кліматом, шторами, охоронною сигналізацією) за розкладом, заданим у календарі. Користувач створює подію у календарі (наприклад, «Кіно 21:00») та прив'язує до неї сцену автоматизації (набір команд для пристроїв), після чого за заданий offset до початку події система самостійно виконує цю сцену.

**Цільова аудиторія:** користувачі розумного будинку, які прагнуть інтегрувати свої щоденні ритуали з фізичним середовищем мешкання без ручного перемикання станів кожного пристрою.

**Особливості:**
- інтеграція з гетерогенними IoT-екосистемами (у майбутньому — MQTT, Matter, Home Assistant API; у поточному інкременті — mock);
- робота у режимі офлайн з локальною синхронізацією;
- мінімалістичний інтерфейс без перевантаження користувача рекламою та підписками;
- орієнтація на приватність — усі дані за замовчуванням зберігаються локально.

## 3. Use Cases та актори

### Актори
- **Користувач** — узагальнений актор, базовий клас для Власника / Члена родини / Гостя.
  - **Власник будинку** — повний доступ.
  - **Член родини** — обмежений доступ, успадковує Користувача.
  - **Гість** — тимчасовий доступ, успадковує Користувача.
- **Адміністратор** — окремий актор, керує ролями.
- **Планувальник** (системний актор) — фоновий процес, що ініціює сцени.
- **IoT-пристрій** (системний актор) — отримує команди та повертає телеметрію.
- **Зовнішня IoT-екосистема** (системний актор) — у майбутньому скоупі.

### Прецеденти (use cases)
1. **Увійти в систему** — Користувач, Адміністратор
2. **Керувати подіями** (CRUD) — Користувач
3. **Налаштувати нагадування** — Користувач (`«extend»` від «Керувати подіями»)
4. **Створити сценарій** — Власник
5. **Виконати сценарій** — Власник, Планувальник, IoT-пристрій
6. **Керувати пристроєм** — Власник, IoT-пристрій
7. **Керувати ролями** — Адміністратор

Усі акторам видно «Увійти в систему», «Керувати подіями» через узагальнення.

## 4. Доменна модель

Доменні класи (`SmartCalendar.Domain`):

### User
```
- Id : Guid
- Email : string
- Name : string
- Role : UserRole  (enum: Owner | Member | Guest | Admin)
- PasswordHash : string
+ Login() : Token
+ CreateEvent(e)
```

### Event
```
- Id : Guid
- Title : string
- StartTime : DateTime
- EndTime : DateTime
- IsRecurring : bool
- RecurrencePattern : string
+ AddReminder(r)
+ IsActiveAt(dt) : bool
```
Композиція 1..* з Reminder. Композиція 1..* з Schedule.

### Reminder
```
- Id : Guid
- OffsetMinutes : int
- IsSent : bool
+ Trigger() : void
+ MarkSent() : void
```

### Scene
```
- Id : Guid
- Name : string
- Commands : List<Command>
+ Execute() : void
+ AddCommand(c)
```
Композиція 1..* з Command. Зв'язок 1..* з Device через Command.

### Schedule
```
- TriggerOffsetMin : int
- IsEnabled : bool
+ Fire() : void
```
Зв'язує Event з Scene (Event 1 — * Schedule * — 1 Scene).

### Device
```
- Id : Guid
- Name : string
- Location : string
- IsOnline : bool
+ SendCommand(c)
+ ReadState() : State
```

## 5. Архітектурні шари

Clean Architecture з 5 шарами (зсередини назовні):

1. **Domain Layer** (`SmartCalendar.Domain`) — POCO-сутності, інтерфейси репозиторіїв, доменні винятки. Жодних зовнішніх залежностей.
2. **Application Layer** (`SmartCalendar.Application`) — use case-сервіси: `EventCommandService`, `EventQueryService`, `SceneExecutor`, `ReminderDispatcher`. Залежить тільки від Domain.
3. **Infrastructure Layer** (`SmartCalendar.Infrastructure`) — конкретні реалізації: EF Core `DbContext`, `EFEventRepository`, `ConsoleMqttPublisher`, `TimerBasedScheduler` (`BackgroundService`).
4. **API Layer** (`SmartCalendar.Api`) — ASP.NET Core Web API + Blazor Server. Hosting, контролери, Razor-сторінки, DI-конфігурація.
5. **Tests** (`SmartCalendar.Tests`) — xUnit + Moq.

Зв'язки: API → Application → Domain ← Infrastructure (Dependency Inversion).

## 6. Функціональні вимоги поточного інкременту

| ID | Опис | Пріоритет |
|---|---|---|
| F1 | Користувач може створити подію (title, start, end) | Must |

| F3 | Користувач може видалити подію | Must |

| F6 | Користувач може прив'язати сцену до події | Must |

| F9 | Користувач може переглянути журнал виконаних сцен | Should |
| F10 | Користувач може редагувати подію | Should |

## 7. Нефункціональні вимоги

- Час реакції UI ≤ 200 мс на дію користувача.
- Точність спрацьовування таймера сцени ≤ ±2 секунди.
- Покриття модульними тестами ≥ 70 % (за ЛР5).
- Цикломатична складність методу ≤ 10 (за ЛР5).
- Розмір класу ≤ 200 LOC (за ЛР5).
