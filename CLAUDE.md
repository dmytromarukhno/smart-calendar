# CLAUDE.md — контекст проєкту для Claude Code

> **Цей файл Claude Code читає першим у будь-якій сесії.** Він описує, що це за проект, які конвенції коду використовуються та які обмеження потрібно зберігати. Перед будь-якою дією прочитай також `SPEC.md`, `IMPLEMENTATION_SCOPE.md` та `CODE_FRAGMENTS.md` у цій папці.

## Що це за проект

**Назва:** Календар для розумного будинку (Smart Home Calendar)
**Тематика:** Інтернет речей (IoT), споживчі платформи розумного будинку
**Автор:** ст. 6ПІ-24б Марухно Дмитро
**Дисципліна:** Основи програмної інженерії, ВНТУ, 2026
**Призначення системи:** інформаційна система, що поєднує функції планувальника подій з автоматизованим керуванням побутовими IoT-пристроями (освітленням, кліматом, шторами, охоронною сигналізацією) за розкладом, заданим у календарі. Користувач створює подію у календарі (наприклад, «Кіно 21:00») та прив'язує до неї сцену автоматизації, після чого за хвилину до початку події інтер'єр самостійно підготується.

## Реальний тех-стек поточного інкременту

Використовуй **тільки** ці технології. НЕ додавай нічого з колонки «Звіт» нижче.

| Призначення | Реалізація |
|---|---|
| Платформа | .NET 8 |
| Мова | C# 12 |
| Веб-сервер та API | ASP.NET Core Web API (Kestrel) |
| Клієнт | **Blazor Server** (одна веб-сторінка, що хоститься тим самим API-процесом) |
| ORM | Entity Framework Core 8 |
| База даних | **SQLite** (файл `smart-calendar.db` у корені) |
| Планувальник | **`BackgroundService` з `System.Threading.Timer`** |
| IoT-publisher | **Mock-клас `ConsoleMqttPublisher`** (пише в `Console.WriteLine`) |
| Логування | Serilog у консоль |
| Тестування | xUnit + Moq + FluentAssertions |
| Контейнеризація | Docker (опціонально, тільки якщо лишається час) |

## Чого НЕ робити

- НЕ додавай .NET MAUI / MAUI Blazor Hybrid — у поточному інкременті лише Blazor Server.
- НЕ підключай реальний MQTT-брокер (Mosquitto) — лише mock-publisher.
- НЕ підключай Home Assistant API, Matter SDK — це майбутній скоуп.
- НЕ підключай PostgreSQL — лише SQLite.
- НЕ використовуй Quartz.NET — заміни на `BackgroundService` + `Timer`.
- НЕ підключай SignalR — для нагадувань у поточному інкременті пиши в консоль.
- НЕ використовуй `throw new Exception("...")` — лише доменні винятки.
- НЕ повертай `null` з публічних методів — кидай винятки або повертай `Option/Maybe`.
- НЕ використовуй магічні числа — виноси у константи (див. `CODE_FRAGMENTS.md` → `ReminderDefaults`).

## Конвенції коду

- Стиль: C# 12 з nullable reference types увімкненими (`<Nullable>enable</Nullable>`).
- Файл `.editorconfig` лежить у корені. Перед комітом запускай `dotnet format`.
- Назви: класи — іменники (`Event`, `SceneExecutor`), методи — дієслова (`ExecuteAsync`, `AddReminder`).
- Async-методи завжди мають суфікс `Async`.
- Усі публічні методи з I/O — асинхронні, приймають `CancellationToken`.
- Доменні класи — `sealed` за замовчуванням; іммутабельні, де можливо (record-типи або private setters).
- Інверсія залежностей через інтерфейси (`IEventRepository`, `IMqttPublisher`, `ISceneRepository`).
- Принципи з ЛР5 («Чистий код» Р. Мартина): один рівень абстракції на метод, методи ≤ 20 рядків, класи ≤ 200 LOC, цикломатична складність ≤ 10.
- Іменовані константи замість магічних чисел (див. `ReminderDefaults` у `CODE_FRAGMENTS.md`).
- Доменні винятки спадкують від `DomainException`; кожен містить контекст у властивостях.

## Робочий процес з тобою (Claude Code)

1. Спочатку прочитай усі `.md` у цій папці.
2. Підтверди, що ти зрозумів обмеження тех-стеку (`IMPLEMENTATION_SCOPE.md`).
3. Виконуй задачі інкрементно: створи solution → доменний шар → інфраструктура → API → Blazor UI → тести.
4. Після кожного етапу запускай `dotnet build` та `dotnet test`.
5. Перед тим, як додавати нову залежність NuGet — питай.
6. Якщо щось у моєму проханні суперечить цьому файлу — звертай мою увагу на конфлікт.
